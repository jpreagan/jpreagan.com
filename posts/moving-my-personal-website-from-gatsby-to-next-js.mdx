As a developer, I'm an advocate for having a personal website. Social media is a great way to grow professionally, but while you're doing so it is also wise to maintain your own platform.

With your own platform, you own all of the content there. You get to control every aspect of its presentation.

For the price of a couple of cappuccinos a year you can acquire your own domain. A personal website will increase your discoverability, brand awareness, and give you more professional credibility.

## Your platform is a training ground to learn new things

There is also one more great benefit to having a personal website: the ability to use it as a training ground. The best way to learn a new framework or technology is to put it into practice!

And why not do that with your own website? Maybe you want a portfolio or to start a blog?

I first wrote my website with plain HTML and CSS. Then React using the [Gatsby](https://www.gatsbyjs.com/) framework, which at the same time gave me exposure to GraphQL on the front end. Then I transitioned that same codebase to TypeScript.

I also used my website to solidify my understanding of static testing analysis tools (e.g. ESLint and prettier), and how to put together a decent front end build system. I [wrote software tests](https://youtu.be/G86Qcdws6E0) with Jest and Cypress.

I've deployed my website with AWS, Netlify, and Vercel. In each instance, you stand to learn something new and acquire valuable new professional skills. This process is wonderfully incremental and you just keep growing while you explore.

## Why I moved on from Gatsby

The previous iteration of my website was built with the Gatsby framework. I'd like to discuss some of the pro's and con's of Gatsby according to my own experience.

Gatsby is a React framework for creating websites and web applications. Gatsby's creator, Kyle Matthews, has described it as a "web compiler."

Gatsby retrieves data from any number of sources and pulls things together for you while making available an internal GraphQL API. It handles page-based routing and does static site generation as well as server-side rendering.

Gatsby's focus has been on performance, accessibility, and developer experience. I've found Gatsby very good for these things. In particular, out of the box, I appreciate how easy it is to get started with a blog.

If you're interested in that check out the official [Gatsby Starter Blog](https://github.com/gatsbyjs/gatsby-starter-blog). You'll get fantastic static site generation with perfect search engine optimization: properly set meta tags and [microdata](https://html.spec.whatwg.org/multipage/microdata.html#introduction-7), a sitemap, and RSS feed. These are things that are very doable in other frameworks too, but you'll likely have to do a bit more work to get them to Gatsby's starting point.

Image optimization with Gatsby is best in class. You bring the image and Gatsby will give you a set of multiple format (JPEG, PNG, WebP and AVIF) responsive images. There are methods to improve the performance of both above-the-fold and below-the-fold images.

I've found every image API with these frameworks has its quirks, especially in regards to styling, but you don't have to fight the API too much with Gatsby.

Gatsby's support for TypeScript is also A+. In addition, the Gatsby documentation is extensive and well-written.

Sounds great so far? But what are the downsides? I would say this is definitely the build time.

Even for a simple site like mine, I was getting several minute build times in production with a couple of minutes best case scenario. This should not be the case when it comes to only a small number of blog posts, a few images, and text.

I'm inclined to agree with Ryan Dahl that [this sort of thing should happen in seconds](https://www.youtube.com/watch?v=pBcFJmQ6UVM) and I also agree we should be moving towards a solution rendering just in time from the edge.

One could only imagine if you had hundreds or thousands of posts, we'd be looking at some substantial build time. [Deferred Static Generation](https://www.gatsbyjs.com/blog/deferred-static-generation-guide/) is the best solution to this particular case though otherwise making even small edits a build nightmare.

The problem has got to the point that Gatsby is nearly unusable on platforms other than Gatsby Cloud, which is the officially supported infrastructure for deploying Gatsby websites and applications.

I could not get Gatsby serverless functions to work on Vercel, got them working on Netlify, but the build time and performance was sluggish. The performance is better on Gatsby Cloud, but the build times are already pretty long in my opinion only likely to get worse as the site grows.

## Moving to Next.js

I love Next.js and Vercel! Converting my website over was a pleasure. As of Next.js version 12, the Next.js compiler is [written in Rust and uses SWC](https://nextjs.org/docs/advanced-features/compiler).

⚡ The first thing I noticed was it's lightning fast. I'm inclined to agree with Lee Robinson that [Rust is the future of JavaScript infrastructure](https://leerob.io/blog/rust).

Even though most front end developers are unfamiliar with Rust, the demand for reliable and efficient build tools is apparent. Rust is the perfect solution for that need, and I'm glad Next.js is going that route too.

Since I'm moving from one React framework to another, converting my small site over wasn't too difficult. I had to ditch using the internal GraphQL API from Gatsby, something that I miss.

The rendering options with Next.js are superb and quite reliable at this point. I improved my Time to First Byte (TTFB) using server-side rendering by about 50% by switching from Gatsby on Gatsby Cloud versus Next.js on Vercel using the same code.

I've some pretty basic needs for my website at present:

- A way to turn a collection of markdown files into blog posts
- Syntax highlighting for code and ideally custom code blocks via MDX
- Page-based routing with server-side rendering a few pages
- Good search engine visibility, sitemap, and RSS feed

I've in mind to expand the personal dashboard, add a light/dark mode, use Edge functions, implement [webmentions](https://www.w3.org/TR/2017/REC-webmention-20170112/), and whatever else seems fun to play with in the near future.
