## Database design and schema creation

In this section, we'll design the database schema for our the API and set up a local MySQL database using Docker Compose. This will help us quickly get a development database up and running without the need to install and configure MySQL directly on your computer.

### Creating the schema

Before setting up the local MySQL database using Docker Compose, let's define the `authors` table for our database. Save the SQL schema to a file named `schema.sql` in the project directory. The schema should look like this:

```sql
CREATE DATABASE IF NOT EXISTS quotd;

USE quotd;

CREATE TABLE authors (
    id INT NOT NULL AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL
);
```

This file creates a `quotd` database if it doesn't exist, sets it as the active database, and defines an `authors` table with an `id` and a `name` column.

### Setting up Docker Compose for MySQL

To set up a local MySQL database using Docker Compose, ensure you have Docker and Docker Compose installed on your computer. You can grab them from the [official Docker website](https://www.docker.com/products/docker-desktop/) if you need to install them.

First, create a new file named `.env` in your project directory and add the following content:

```
MYSQL_ROOT_PASSWORD=root
MYSQL_DATABASE=quotd
MYSQL_USER=mysql
MYSQL_PASSWORD=mysql
MYSQL_HOST=localhost
MYSQL_PORT=3306

DSN=${MYSQL_USER}:${MYSQL_PASSWORD}@tcp(${MYSQL_HOST}:${MYSQL_PORT})/${MYSQL_DATABASE}
```

This `.env` file stores the necessary environment variables for your MySQL container. Adding the `.env` file to your `.gitignore` is a good idea to prevent accidentally committing sensitive information to your version control system. To do this, run the following command:

```sh
touch .gitignore && echo .env >> .gitignore
```

This command appends `.env` to your `.gitignore`, ensuring that the file isn't tracked by Git.

Next, create a new file named `docker-compose.yml` in your project directory with the following content:

```yaml
services:
  db:
    image: mysql:8
    restart: always
    env_file:
      - .env
    ports:
      - "3306:3306"
    volumes:
      - ./schema.sql:/docker-entrypoint-initdb.d/schema.sql
      - ./mysql-data:/var/lib/mysql
```

This configuration file sets up a MySQL 8 container with the required environment variables for the root password, database name, and user credentials. It also maps the container's port 3306 to the host machine's port 3306, mounts the `schema.sql` file to be executed on container startup, and binds the host directory `./mysql-data` to the container's `/var/lib/mysql` directory for data persistence.

To start the MySQL container, run the following command in your terminal from the project directory:

```sh
docker-compose up -d
```

This command downloads the MySQL image (if you don't have it already), creates the container, and starts it in the background. The mounted `schema.sql` file will be executed when the MySQL container starts, creating the `authors` table in the `quotd` database.

By binding the host directory `./mysql-data` to the container's `/var/lib/mysql` directory, your data will persist even when you bring the Docker service down and up again. The `schema.sql` script will still run on container startup, but since the data directory is no longer empty, the MySQL container will skip executing it, thus preserving your existing data.

You can now connect to your MySQL database using the provided credentials at `localhost:3306`. Should you need to interact directly with the database, use the following command:

```sh
docker-compose exec db mysql -u mysql -p
```

When prompted, enter the password specified in the `.env` file.

## SQL queries and sqlc

In this section, we'll write SQL queries for working with authors and use sqlc to generate Go code for interacting with our MySQL database. sqlc is a powerful tool that takes SQL queries as input and generates type-safe Go code, reducing the risk of errors and making it easier to work with databases in Go.

### Writing SQL queries

First, let's write the SQL queries for our authors table. We'll need queries to:

- Get an author by ID
- List all authors
- Create an author
- Delete an author

Save these queries in a file named `query.sql` in the project directory:

```sql
-- name: GetAuthor :one
SELECT \* FROM authors
WHERE id = ? LIMIT 1;

-- name: ListAuthors :many
SELECT \* FROM authors
ORDER BY name;

-- name: CreateAuthor :execresult
INSERT INTO authors (name)
VALUES (?);

-- name: DeleteAuthor :exec
DELETE FROM authors
WHERE id = ?;
```

These queries use sqlc's special comment syntax to specify each query's name, type, and behavior.

### Installing sqlc

To install sqlc globally, run the following command in your terminal:

```sh
go install github.com/kyleconroy/sqlc/cmd/sqlc@latest
```

This command will download the latest version of sqlc and install it in your `$GOPATH/bin` directory. Ensure this directory is included in your `PATH` so you can run the `sqlc` command from anywhere.

### Configuring sqlc

To configure sqlc, create a file named `sqlc.yaml` in the project directory with the following content:

```yaml
version: 1
packages:
  - path: "repository"
    name: "repository"
    engine: "mysql"
    schema: "schema.sql"
    queries: "query.sql"
    emit_json_tags: true
```

This configuration tells sqlc to generate a package named `repository` in the `./repository` directory, using MySQL as the engine, and using the `schema.sql` and `query.sql` files for schema and queries. Setting `emit_json_tags` to `true` will automatically generate JSON struct tags for the model fields, resulting in the generally more desirable lowercase field names in the JSON output.

### Generating Go code with sqlc

Now that sqlc is installed and configured, we can generate Go code from our SQL queries. Run the following command in your project directory:

```sh
sqlc generate
```

This command will create a new `./repository` directory in your project root and generate a `repository.go` file containing Go code for interacting with the `authors` table. The generated code is type-safe and provides a simple API for executing the SQL queries we defined earlier.

Your project directory should now look like this:

```
.
├── docker-compose.yml
├── go.mod
├── handlers
├── mysql-data
│ └── ...
├── query.sql
├── repository
│ └── repository.go
├── schema.sql
└── sqlc.yaml
```

With the SQL queries written and the Go code generated using sqlc, we're ready to move on to the next section, where we'll implement the API endpoints and handlers.

## Implementing API endpoints and handlers

In this section, we'll implement the API endpoints and handlers for working with authors using the Gin web framework and the generated repository code from the previous section. We'll create handlers for getting an author by ID, listing all authors, creating an author, and deleting an author.

### Setting up the Gin web framework

Gin is a lightweight web framework for Go that makes it easy to build HTTP services with minimal boilerplate. It provides a flexible routing system, middleware support, and convenient methods for handling JSON and other types of requests and responses.

We'll need to install Gin, and a couple of other packages with `go get`:

```sh
go get github.com/gin-gonic/gin
go get github.com/go-sql-driver/mysql
go get github.com/joho/godotenv
```

The MySQL driver is needed for database access, and the `godotenv` package will be used to load environment variables from the `.env` file.

### Creating the application entry point

First, let's write the `main.go` file, which will serve as the entry point for our application. In the root directory of your project, create a `main.go` file with the following content:

```go
package main

import (
	"database/sql"
	"log"
	"os"

	"github.com/gin-gonic/gin"
	_ "github.com/go-sql-driver/mysql"
	"github.com/joho/godotenv"
	"github.com/jpreagan/quotd-api/handlers"
	"github.com/jpreagan/quotd-api/repository"
)

var db *sql.DB

func main() {
	err := godotenv.Load()
	if err != nil {
		log.Fatal("failed to load env", err)
	}

	db, err = sql.Open("mysql", os.Getenv("DSN"))
	if err != nil {
		log.Fatal("failed to connect to database", err)
	}

	router := gin.Default()
	queries := repository.New(db)
	handlers.SetupRoutes(router, queries)
	router.Run()
}
```

This code sets up a new Gin router, connects to the MySQL database using the `DSN` environment variable, and initializes the generated repository with the database connection. It then calls the `handlers.SetupRoutes` function to set up the API routes and start the HTTP server.

### Creating handlers

Next, let's create the handlers for our API endpoints. Handlers in Gin are functions that process incoming HTTP requests and generate appropriate responses. To make it easier for those new to Gin, we will briefly overview how handler functions work in this framework.

In the handlers directory, create a new file named handlers.go with the following content:

```go
package handlers

import (
	"github.com/gin-gonic/gin"
	"github.com/jpreagan/quotd-api/repository"
)

func SetupRoutes(router *gin.Engine, queries *repository.Queries) {
	router.GET("/authors", getAuthorsHandler(queries))
	router.GET("/authors/:authorId", getAuthorHandler(queries))
	router.POST("/authors", createAuthorHandler(queries))
	router.DELETE("/authors/:authorId", deleteAuthorHandler(queries))
}
```

This file defines a `SetupRoutes` function that sets up the API routes for getting an author by ID, listing all authors, creating an author, and deleting an author. Each route is associated with a corresponding handler function, which we'll implement next.

In the `/handlers` directory, create a new file named `authors.go` with the following content:

```go
package handlers

import (
	"context"
	"net/http"
	"strconv"

	"github.com/gin-gonic/gin"
	"github.com/jpreagan/quotd-api/repository"
)

func getAuthorsHandler(queries *repository.Queries) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := context.Background()
		authors, err := queries.ListAuthors(ctx)
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}
		c.JSON(http.StatusOK, authors)
	}
}

func getAuthorHandler(queries *repository.Queries) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := context.Background()
		authorId, err := strconv.Atoi(c.Param("authorId"))
		if err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": "Invalid author ID"})
			return
		}
		author, err := queries.GetAuthor(ctx, int32(authorId))
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}
		c.JSON(http.StatusOK, author)
	}
}

func createAuthorHandler(queries *repository.Queries) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := context.Background()
		var newAuthor repository.Author
		if err := c.BindJSON(&newAuthor); err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": "Invalid request body"})
			return
		}
		result, err := queries.CreateAuthor(ctx, newAuthor.Name)
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}
		id, err := result.LastInsertId()
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": "Failed to retrieve last insert ID"})
			return
		}
		newAuthor.ID = int32(id)
		c.JSON(http.StatusCreated, newAuthor)
	}
}

func updateAuthorHandler(queries *repository.Queries) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := context.Background()
		authorID, err := strconv.Atoi(c.Param("authorId"))
		if err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": "Invalid author ID"})
			return
		}

		var updatedAuthor repository.Author
		if err := c.BindJSON(&updatedAuthor); err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": "Invalid request body"})
			return
		}

		updatedAuthor.ID = int32(authorID)
		err = queries.UpdateAuthor(ctx, repository.UpdateAuthorParams{
			Name: updatedAuthor.Name,
			ID:   int32(authorID),
		})
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}

		author, err := queries.GetAuthor(ctx, int32(authorID))
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}

		c.JSON(http.StatusOK, author)
	}
}

func deleteAuthorHandler(queries *repository.Queries) gin.HandlerFunc {
	return func(c *gin.Context) {
		ctx := context.Background()
		authorId, err := strconv.Atoi(c.Param("authorId"))
		if err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": "Invalid author ID"})
			return
		}
		err = queries.DeleteAuthor(ctx, int32(authorId))
		if err != nil {
			c.JSON(http.StatusInternalServerError, gin.H{"error": err.Error()})
			return
		}
		c.JSON(http.StatusOK, gin.H{"message": "Author deleted successfully"})
	}
}
```

The above code defines handler functions for each of the API endpoints associated with authors. Each handler function takes a `*repository.Queries` argument, which allows it to interact with the database, and returns a `gin.HandlerFunc`. The `gin.HandlerFunc` is responsible for processing the request, such as validating input, fetching data, or rendering a response.

Remember that each handler function is associated with a specific route. So, when a request comes in, Gin will match the request's URL and HTTP method to a route and invoke its associated handler function. This modular approach to handling requests makes it easy to add, modify, or remove functionality as needed.

## Improving Error Handling with Custom Types

In this section, we'll improve error handling in our handlers by defining custom error types. This will help us provide more meaningful error messages and make it easier to maintain our code as it grows.

### Why better error handling is necessary

When building APIs, providing meaningful and consistent error messages is essential to help clients understand and handle errors correctly. By default, Gin returns generic error messages that might not be informative enough for API clients. Creating custom error types allows us to return more specific error messages and status codes, making it easier for clients to understand the cause of the error and take appropriate action.

### Defining custom error types

Let's now define custom error types for better error handling and provide our client with more informative error messages. To do this, we will create a new package called `errors`. This separation will help keep our code modular and maintainable.

First, create a new directory called `/errors` in your application's root directory. This is where we will store our custom error types.

Next, create a new file inside the `/errors` directory called `errors.go`. In this file, we will define our custom error types. Add the following code to `errors.go`:

```go
package errors

import "net/http"

type APIError struct {
	Code    int    `json:"code"`
	Message string `json:"message"`
}

func NewAPIError(code int, message string) *APIError {
	return &APIError{Code: code, Message: message}
}

// Common errors
var (
	ErrInvalidID          = NewAPIError(http.StatusBadRequest, "Invalid ID")
	ErrInvalidRequestBody = NewAPIError(http.StatusBadRequest, "Invalid request body")
)

// Author errors
var (
	ErrAuthorNotFound = NewAPIError(http.StatusNotFound, "Author not found")
)

// Quote errors
var (
	ErrQuoteNotFound = NewAPIError(http.StatusNotFound, "Quote not found")
)

// Category errors
var (
	ErrCategoryNotFound = NewAPIError(http.StatusNotFound, "Category not found")
)
```

In this code snippet, we define a new struct called `APIError` that will hold our custom error information. We also create a helper function `NewAPIError` to create instances of `APIError` easily. We then define several common, author, quote, and category errors as package-level variables that can be used throughout our application.

Using custom error types, we standardize the structure of our error responses, making it easier for clients to understand and handle them. It also helps keep our code more organized and maintainable, as we can quickly identify and handle specific error cases.

Next, we'll update the handlers in the `handlers/authors.go` file to use the custom error types we just defined. Make sure to import the errors package at the beginning of the file:
